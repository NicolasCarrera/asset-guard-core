# AGENTS.md — asset-guard-core

## Project overview

**asset-guard-core** is a Java 25 / Spring Boot 4.x REST API for Fixed Asset Management (EAM/ITAM).
It handles serialized assets (vehicles, machinery, IT hardware), custody workflows,
lifecycle tracking, maintenance records, and depreciation calculations.

- **Language / platform:** Java 25, Spring Boot 4.x (target 4.0.3), Maven
- **Base package:** `com.assetguard.core`
- **Entrypoint:** `com.assetguard.core.Application`

## Architecture

Follows a **classic layered architecture**: Controller → Service → Repository → Model.
Each layer has a single responsibility and depends only on the layer below it.

```
com.assetguard.core
├── controller/    # REST endpoints — HTTP in, HTTP out, nothing else
├── service/       # Business logic, orchestration, transaction boundaries
├── repository/    # Spring Data JPA interfaces + custom queries
├── model/         # JPA entities (persisted state)
├── dto/           # Request/response objects for HTTP and service boundaries
├── mapper/        # MapStruct interfaces — model ↔ dto conversions
├── exception/     # Domain exceptions + @RestControllerAdvice handler
└── config/        # Spring beans, security, Flyway, OpenAPI configuration
```

**Layer rules — enforce always:**
- Controllers call services only. No repository calls, no JPA, no business logic.
- Services own all business rules and transaction boundaries (`@Transactional`).
- Repositories are interfaces (Spring Data JPA). Custom JPQL/native queries go in `*RepositoryCustom`.
- Models (entities) are never exposed in HTTP responses. Always map to a DTO first.
- Mappers are generated by MapStruct. Never write manual mapping code.
- Exceptions are defined once in `exception/` and mapped to HTTP status codes in a single `@RestControllerAdvice`.

## Java 25 and Spring Boot 4 — modern syntax rules

The project targets **Java 25** and **Spring Boot 4.x**. Always use modern language features.
Do not write pre-Java 17 style code under any circumstance.

### Records for DTOs and value objects

Use `record` for any object that carries data without mutable state.
Records are immutable by default, which eliminates accidental mutation bugs and makes
objects safe to pass across service boundaries and threads without defensive copies —
critical in distributed and concurrent systems.

```java
// Correct
public record AssetRequest(@NotBlank String assetTag, @NotBlank String description) {}

// Wrong — do not use mutable classes for DTOs
public class AssetRequest { private String assetTag; ... }
```

### Sealed classes for closed type hierarchies

Use `sealed` + `permits` for states, result types, or any hierarchy with a fixed set of subtypes.
The compiler guarantees exhaustiveness in `switch` expressions, making missing-case bugs impossible.

```java
public sealed interface AssetEvent permits AssetCreated, AssetVoided, AssetDecommissioned {}
```

### Pattern matching and switch expressions

Use `switch` expressions (not statements) with pattern matching.
Avoid `if/else` chains and raw `instanceof` casts.

```java
// Correct
String label = switch (status) {
    case ACTIVE         -> "In use";
    case MAINTENANCE    -> "Under maintenance";
    case DECOMMISSIONED -> "Retired";
};

// Pattern matching instanceof — no explicit cast needed
if (event instanceof AssetVoided voided) {
    log.info("Voided: {}", voided.reason());
}
```

### Text blocks for multiline strings

Use text blocks for JPQL, SQL snippets, and multiline messages. Never concatenate strings.

```java
String jpql = """
        SELECT a FROM Asset a
        WHERE a.status = :status
          AND a.assetTag = :tag
        """;
```

### `Optional` — no null returns from public methods

Services and repositories must never return `null`. Use `Optional<T>` for values that may be absent.

```java
// Repository
Optional<Asset> findByAssetTag(String tag);

// Service
assetRepository.findByAssetTag(tag)
    .orElseThrow(() -> new AssetNotFoundException(tag));
```

### Immutability — prefer it everywhere

Immutable objects have no hidden shared mutable state, require no synchronization,
and produce deterministic audit trails — all essential in an asset management system
where data correctness and traceability are non-negotiable.

- Use `record` for DTOs and value objects.
- Use `final` fields in JPA entities wherever JPA allows it.
- Use `List.of()`, `Map.of()`, `Set.of()` for constants and static collections.
- Never expose the internal collection of an entity directly; return `List.copyOf(...)`.

### Virtual threads (Spring Boot 4 default)

Spring Boot 4 enables virtual threads by default (`spring.threads.virtual.enabled=true`).
Do not reach for `CompletableFuture` or manual thread pools for standard blocking I/O — virtual threads handle this transparently.

### `@HttpExchange` for declarative HTTP clients

When integrating with external HTTP APIs, use `@HttpExchange` (Spring 6+) interfaces
instead of `RestTemplate` or manually configured `WebClient`.

### Constructor injection — always

Never use field injection (`@Autowired` on a field). Always use constructor injection.
Use Lombok `@RequiredArgsConstructor` to eliminate the boilerplate.

```java
@Service
@RequiredArgsConstructor
public class AssetService {
    private final AssetRepository assetRepository;
    private final AssetMapper      assetMapper;
}
```

## Database

- Database: **PostgreSQL**
- Migrations: Flyway — files in `src/main/resources/db/migration/`, named `V<version>__<description>.sql`
- Required environment variables:

```
SPRING_DATASOURCE_URL=jdbc:postgresql://localhost:5432/assetguard
SPRING_DATASOURCE_USERNAME=asset
SPRING_DATASOURCE_PASSWORD=secret
SPRING_PROFILES_ACTIVE=local
```

- **Never hard-delete assets.** Use `status` field transitions only. Envers provides the immutable audit trail.
- All JPA entities must extend `AuditMetadata` (`@MappedSuperclass` in `model/shared/`)
  which holds `createdAt`, `updatedAt`, `createdBy`, `updatedBy` — populated automatically by `@EnableJpaAuditing`.

## Business rules — enforce in the service layer

| ID | Rule |
|---|---|
| RNI-01 | `assetTag` is globally unique — even for voided/decommissioned assets. |
| RNI-02 | A component can have at most one parent asset at a time. |
| RNI-03 | Custody transfers are 1:1 — only one active custody per asset at any time. |
| RNI-04 | Custody acceptance requires explicit confirmation (signature / acceptance flag). |
| RNI-05 | Physical deletion is forbidden. Valid terminal states: `VOIDED`, `DECOMMISSIONED`. |
| RNI-07 | Assets in `MAINTENANCE` or `CRITICAL_FAILURE` state cannot be assigned. |
| RNI-08 | Maintenance interventions must record technician, date, and cost. |
| RNI-09 | Depreciation must never reduce NBV below the residual value. |
| RNI-10 | Capitalizable improvements update NBV without deleting historical depreciation records. |

Write a unit test for every RNI rule. Tag tests with `@Tag("business-rule")`.

## Code conventions

- **Money / accounting values:** always `BigDecimal`. Never `double` or `float`.
- **HTTP input:** always use a `record`-based DTO annotated with Jakarta Validation. Never expose JPA entities over HTTP.
- **No logic in controllers:** parse HTTP → call one service method → return `ResponseEntity`. Nothing else.
- **Formatter:** Google Java Format. Run `mvn spotless:apply` before every commit.
- **Exceptions:** one exception class per business error in `exception/domain/`. A single `@RestControllerAdvice` in `exception/handler/` maps them to HTTP status codes.
- **Logging:** use Lombok `@Slf4j`. Never log passwords, tokens, personal data, or PII at any log level.

## Testing guidelines

- Unit tests: use Mockito (`@ExtendWith(MockitoExtension.class)`). Cover all service methods and every RNI rule.
- Integration tests: `@SpringBootTest` with a real PostgreSQL container via Testcontainers. Cover repositories and full HTTP cycles.
- All tests must pass before merging: `mvn verify`.

## Security and audit

- All write endpoints (`POST`, `PUT`, `PATCH`, `DELETE`) require authentication (Spring Security).
- Envers audits every entity mutation automatically — never bypass auditing with native queries or direct JDBC.
- Secrets are injected via environment variables only. No hardcoded credentials in source or config files.
- Never log sensitive fields (passwords, personal data, tokens).
